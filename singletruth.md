# Single Source of Truth Map

| Component | Path | What It Centralizes | Primary Consumers / Notes |
|-----------|------|----------------------|---------------------------|
| Engineering contract | `LLM_CODING_PROMPT.md` | Defines the required Clean Architecture layout, SOLID rules, and cross-cutting conventions (error wrapping, context propagation, DI). Keeps every repo aligned on layering so business rules stay isolated. | All Go services, BFF, and even frontend build scripts reference this as the process authority when adding code. |
| Service bootstrap & auth helpers | `go-kit/{auth,errs,jwt-key-selector,uber-fx}` | Centralized JWT parsing, auth context propagation, structured error helpers, JWKS selection, and Fx modules for gRPC/postgres/telemetry. Ensures every service trusts the same token validation and instrumentation stack. | Every Go microservice imports these modules for HTTP/gRPC interceptors, DB wiring, tracing, and uniform error semantics. |
| Account identities | `account-management-service/app/internal/core/account` | Domain models and services for user registration, login, and role assignment; integrates with Zitadel. This is the truth for how Chirp identities are created and linked to org/user-profile records. | BFF (auth), organisation-service (member linkage), user-profile-service (profile bootstrap) and any client trusting issued JWTs. |
| Organization structure & rights | `organisation-service/app/internal/domain` + `organisation-service/app/internal/repository/accesscontrol` | Canonical org aggregates plus Casbin-backed policy storage that define membership, predefined roles, invitations, and ownership transfers. | Resource Inventory, Subscription (limits per org), Notification, BFF, and Account Management rely on this to resolve user rights. |
| User data | `user-profile-service/pkg/domain` | Canonical Mongo-backed representation of profile fields, preferences, wallets, and notification quotas. | Account Management (profile enrichment), Notification Service (quota checks), BFF (user settings), Frontend (user dashboards). |
| Resource inventory | `resource-inventory-service/app/internal/domain` | Definitive schemas for devices, SIMs, connectors, dashboards, sharing/access rules. Stores authoritative inventory claims per customer. | BFF (devices, SIMs, dashboard views), Subscription (resource limits), Automation engines (device metadata), Frontend modules (Devices, Sims, Gateways). |
| Subscription plans & limits | `subscription/app/internal/domain` | Models checkout sessions, Stripe customers, plan rules, device count limits, and validation routines. Only place where paywall logic and entitlements are defined. | Resource Inventory feature flags, BFF subscription endpoints, Billing flows on frontend, Notification throttling logic. |
| Notification semantics | `notification-service/app/internal/domain` + `notification-service/templates` | Canonical template catalog, notification payload contracts, verification tokens, alert histories. Provides authoritative wording/layout plus quota rules. | User Profile (limits), BFF (alert center), Automation (rule & workflow triggers), Supersend integration. |
| Tenant access enforcement | `organisation-service/app/internal/service/accesscontrol` + `resource-inventory-service/app/internal/integration/organization` + `bff/pkg/integration/organizations` | Enforces tenant isolation via the CheckAccess gRPC contract backed by Casbin policies; downstream services must call it before manipulating org-scoped resources. | Resource Inventory, BFF, Subscription, Notification, and other services gate their operations by invoking these APIs with the user/org context. |
| Shared backend constants | `go-kit/{auth,errs}` & each service’s `internal/domain` packages | Authentication claim constants, error codes, domain enums, and feature-flag keys that backend services reuse instead of hardcoding. | All Go microservices import these when defining transport responses or domain validation logic. |
| Device events & calendars | `event-service/app/internal/domain` + `event-service/app/internal/repository/{storage,normalized}` | Canonical ingestion and storage of device events (JetStream ingress, Timescale persistence) and calendar summaries. | BFF (event APIs), Automation (rule evaluation), Inventory dashboards, Frontend charts/feeds. |
| Rule definitions & versions | `rule-modeler-engine/app/internal/domain` | Authoritative storage for rule definitions, versions, and BPMN diagrams with collaborative WebSocket-based locking, version control (save/restore/rename), and subscription-enforced rule limits. Manages the full design-time lifecycle of rules before execution. | BFF (rule designer endpoints), Frontend rule designer UI, rules-engine-executor (consumes finalized rules for execution), Automation service. |
| Rule execution runtime | `rules-engine-executor/app/internal/domain` | Runtime execution engine for finalized rules from rule-modeler-engine. Currently a boilerplate skeleton (based on go-boilerplate) with gRPC + NATS JetStream patterns ready for rule execution domain logic. | Rule Modeler Engine (provides rule definitions), BFF (execution endpoints), Automation service (orchestration), Event service (device events as rule inputs). |
| Automation execution & runtime | `automation-service/app/internal/core/{rule,workflow}` + `automation-service/app/pkg/proto` (v1/v2) | Runtime execution of rules, timers, actions, and workflows via Temporal/Gorules; handles active rule instances and their state. | BFF (automation endpoints), Frontend automation UI, Inventory/Events as data providers, Notification for outputs. |
| Data credits & payments | `data-credits-service/app/internal/service` + `data-credits-service/app/internal/infrastructure/db/migrations` | Authoritative balances and payment state (credits ledger, payments, IPN updates) backed by Postgres and Sui/NowPayments integrations. | BFF billing flows, Frontend wallet/balance UIs, Subscription entitlements, Reporting. |
| UI design system | `ui-kit/src/{lib,styles}` | Shared React components, interaction patterns, design tokens, and CSS variables—single place to change look & feel. | `chirp-frontend` and any partner apps import this package to guarantee identical UX behavior. |
| Frontend locales | `chirp-frontend/src/locales` | Translation dictionaries for all supported languages; keeps copy consistent across modules. | `chirp-frontend` pages/modules load keys from here for i18n; other apps reuse them via shared packages when needed. |
| Frontend constants | `chirp-frontend/src/const` | Master list of UI-visible constants (device types, theme tokens, URLs, wallet copy, widget definitions) so features never hardcode strings. | All React modules/features under `chirp-frontend/src/modules`, plus hooks and services referencing repeated labels/URLs. |
| Frontend domain contracts | `chirp-frontend/src/entities` | TypeScript definitions for every backend concept (devices, sims, notifications, workflows, payments). Drives forms, state slices, and API typing so UI logic shares a single schema. | All React modules/features under `chirp-frontend/src/modules` plus hooks/services to keep view models consistent. |
| Unified API facade | `bff/pkg/api` + `bff/pkg/service` | BFF enforces a single public API surface that composes ChirpStack + internal microservices into cohesive responses. Prevents frontend from calling microservices directly and keeps mapping logic centralized. | `chirp-frontend` consumes only BFF endpoints; automation scripts and external clients also rely on this consistent facade. |
